TEST 37 (FULL 4-CONDITION BENCHMARK - REVEALING THE PROBLEM)

Scope: Balanced preset, 16 fixtures, 3 runs per model, testing all 4 RAG x Thinking conditions.

Key finding: Your research notes on RAG failure modes were exactly right. RAG integration is currently degrading performance instead of helping. For example, gpt-oss without RAG achieves 50.7% F1, but with RAG it drops to 40.7% (a 10 percentage point loss).

Performance without RAG (baseline):
- gpt-oss: F1 50.7%, Precision 69.9%, but high false negatives (misses 674 issues)
- kimi-k2.5: F1 42.0%, Recall 62.0%, but high false positives (hallucinates 399 issues)
- qwen3.5: F1 47.9%, but slowest (~293 seconds per run)

The problem: Each model struggles with different dimensions. gpt-oss has high recall but misses real issues. kimi has high recall but reports many false alarms. Neither is balanced alone. And crucially, RAG is making things worse, not better.

TEST 38 (POST-HYBRID RAG FIXES - ADDRESSING RETRIEVAL ISSUES)

Based on your feedback about retrieval failures, context overload, and generation issues, I implemented three fixes:

1. Hybrid Search (BM25 + Semantic Embeddings)
   - Installed rank-bm25 library
   - Updated app.py to build BM25 index during indexing
   - Rewrote retrieve endpoint with 4-stage hybrid logic:
     Stage 1: Semantic search (ChromaDB vector similarity)
     Stage 2: BM25 keyword search
     Stage 3: Merge, deduplicate, score combined results
     Stage 4: Return top-k sorted by combined score

2. Reduced Context Overload
   - Changed top_k chunks: 5 to 3 per query
   - Reduces noise and hallucinatory responses

3. Improved Grounding Instructions
   - Better prompting for when models should use context vs internal knowledge

ROOT CAUSE - THE FUNDAMENTAL ISSUE

Your research notes were spot-on. The issue isn't just chunking or hybrid search—it's fundamentally that our KB is organized by accessibility concept (forms.md, images.md, tables.md) but fixtures ask questions about components (navbar, form-field, header).

When a fixture asks "analyze this navbar," the RAG retrieves chunks about "ARIA landmarks" and "keyboard navigation" (conceptually right but structurally wrong). The model gets confused trying to map generic accessibility rules to specific component code.

Hybrid search helps the models reason better (rag-think +4pp) but doesn't fix the core retrieval mismatch.
