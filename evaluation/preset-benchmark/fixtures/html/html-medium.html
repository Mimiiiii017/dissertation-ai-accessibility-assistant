How Luminary Studio works

Get from installation to your first audit in under five minutes.

  1. 1
    Connect your project

    Install the Luminary CLI (npm install -g @luminary/cli), connect your GitHub or GitLab repository, and point the scanner at your staging URL. Supports single-page apps, server-rendered pages, and native web components.

  2. 2

    Run your first scan

    Execute luminary scan or trigger the scan from inside the VS Code extension. The engine crawls your app, runs 320+ automated checks across all WCAG 2.2 success criteria, and delivers results in real time.

  3. 3

    Review and remediate

    Open the Luminary dashboard to see issues grouped by severity, WCAG criterion, component, and route. Each issue includes a code diff, video walkthrough, and a suggested ARIA or HTML fix.

  4. 4

    Ship and monitor

    Merge the fixes and push to production. Luminary's nightly monitor checks that issues stay resolved, alerts on regressions, and tracks your accessibility score over time so stakeholders always have an up-to-date picture.

Frequently asked questions

Luminary Studio covers WCAG 2.0, 2.1, and 2.2 at levels A, AA, and AAA. We also include checks for EN 301 549 (European standard), Section 508 (US Federal), and the upcoming WCAG 3.0 draft criteria. Our rule set is updated within 30 days of any official standard revision.

You can provide session cookies, OAuth tokens, or a login script that the scanner will execute before crawling. We support cookie-, form-, and token-based authentication. Enterprise plans support SAML and SSO-gated environments.

Luminary only stores scan result metadata (issue descriptions, element selectors, screenshots) — never your source code. All data is encrypted at rest (AES-256) and in transit (TLS 1.3). Enterprise customers can choose EU-only data residency or deploy fully on-premise.
