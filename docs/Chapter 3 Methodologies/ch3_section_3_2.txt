3.2  Scope, Limitations, and Reproducibility
─────────────────────────────────────────────────────────────────────────────────

───────────────────────────────────────────────────────
3.2.1  Domain Scope
───────────────────────────────────────────────────────

The evaluation is restricted to WCAG 2.2 Level A and Level AA success criteria
(SCs) applied to source code written in four web technologies: HTML, CSS,
JavaScript, and TypeScript with React JSX syntax (TSX).  These technologies were
selected because they constitute the four principal surface areas through which
WCAG 2.2 conformance is implemented and violated.  HTML governs the semantic
document structure, landmark roles, and ARIA attribute markup that assistive
technologies depend upon for navigation and comprehension.  CSS controls visual
characteristics with accessibility implications including colour contrast ratios,
focus indicator styling, motion and animation behaviour, and the positioning of
screen-reader utility classes.  JavaScript and TypeScript/TSX manage dynamic ARIA
state transitions (aria-expanded, aria-pressed, aria-invalid), keyboard event
handling, focus management, and live-region announcements — violations that are
entirely invisible to static file scanners.  Level A and Level AA were chosen
as the target conformance levels because they represent the internationally
mandated standards under the European Accessibility Act 2025 and the UK Public
Sector Bodies Regulations 2018.  WCAG 2.2 Level AAA success criteria were
excluded on the grounds that they do not carry mandatory compliance obligations
in most legal jurisdictions and addressing them would have widened the scope
beyond what the evaluation fixtures could validly represent.

The study specifically targets semantic violations — those requiring contextual
or structural reasoning that cannot be reduced to a simple attribute-presence
check.  Deterministic rule-based tools such as axe-core, Pa11y, and WAVE were
assessed as out of scope for use as the primary evaluation instrument because
ground-truth validation conducted as part of this study found a pattern-matching
accuracy of approximately 1.2 per cent against the semantic violation set,
confirming that such tools systematically under-report the category of violation
targeted here.  This finding is consistent with the conclusions of Abou-Zahra
et al. (2023) [14] and Campoverde-Molina et al. (2021) [17], who each
independently identified the inability of automated rule-based checkers to detect
violations requiring semantic understanding.

The knowledge base covers all WCAG 2.2 SCs at Levels A and AA, organised by
technology category (ARIA, controls, dialogs, forms, headings, images, input
modalities, keyboard, media, navigation, structure, tables, timing, visual) and
supplemented by NASA-TLX mapping documents used by the cognitive-load prediction
subsystem.

───────────────────────────────────────────────────────
3.2.2  Experimental Variables
───────────────────────────────────────────────────────

The study employs seven categories of variable, as described below.

Independent Variables (IVs)

The primary independent variables are those whose values were systematically
varied during the benchmarking experiments.

IV1 — RAG Condition.  This binary variable determines whether the WCAG
knowledge-base context is injected into the prompt.  The two levels are RAG
Enabled (top-k relevant chunks retrieved and included) and RAG Disabled (no
retrieval context; the model operates on its pre-trained knowledge alone).
This variable directly operationalises the knowledge-grounding component of H1.

IV2 — Chain-of-Thought (Thinking) Condition.  This binary variable determines
whether the model is permitted to emit extended reasoning before generating its
final response.  The two levels are Thinking Enabled (reasoning models may
produce a <think>...</think> block which is then stripped from the output before
scoring) and Thinking Disabled (the /no_think directive is appended to the
prompt, suppressing this behaviour).  The combination of IV1 and IV2 defines the
four cells of the 2×2 ablation study.

IV3 — LLM Model.  The model family and size constitute a third independent
variable in Phase 1.  Eighteen cloud-hosted models from providers including
Google (Gemini), Moonshot (kimi), Zhipu AI (GLM), Alibaba (Qwen), DeepSeek,
MiniMax, NVIDIA (nemotron), Meta (Llama), Mistral, and OpenAI were evaluated.
Cloud-hosted deployment was adopted following a preliminary feasibility
assessment (Section 3.5.1) in which all four locally deployed Ollama models
failed with connection errors across every condition, making local deployment
unviable as an evaluation infrastructure.  Varying the model whilst holding all
other variables constant enables the selection of the optimal model for
deployment.

Dependent Variables (DVs)

The dependent variables are the measured outcomes produced by the evaluation
framework for each model run.

DV1 — F1 Score.  The harmonic mean of precision and recall, computed against
the manually annotated ground-truth concept pool for each fixture.  F1 serves
as the primary accuracy metric throughout the study.  Formula:
F1 = 2 × Precision × Recall / (Precision + Recall).

DV2 — Matthews Correlation Coefficient (MCC).  A more reliable single-figure
quality metric than F1 for binary classification tasks with imbalanced class
distributions.  Chicco and Jurman (2020) demonstrated through systematic
analysis that MCC is the only binary classification metric that accounts for all
four cells of the confusion matrix proportionally, making it robust under the
severe class imbalance characteristic of accessibility auditing — where the
concept pool of possible violations vastly outnumbers those present in any given
fixture [26].  F1 ignores true negatives entirely and is therefore misleadingly
optimistic when the negative class dominates.  MCC ranges from −1 (perfect
inverse prediction) through 0 (random) to +1 (perfect prediction).
Formula: MCC = (TP·TN − FP·FN) / √[(TP+FP)(TP+FN)(TN+FP)(TN+FN)].

DV3 — Precision.  The proportion of reported violations that are genuine:
Precision = TP / (TP + FP).

DV4 — Recall.  The proportion of ground-truth violations that were detected:
Recall = TP / (TP + FN).

DV5 — Specificity.  The proportion of correct non-detections among all true
negatives: Specificity = TN / (TN + FP).

DV6 — Negative Predictive Value (NPV).  NPV = TN / (TN + FN).

DV7 — NASA-TLX Raw Workload Score.  The predicted cognitive workload associated
with auditing and remediating the violations detected in a given fixture,
estimated computationally using the NASA-TLX prediction model described in
Section 3.5.5.

Controlled Variables (CVs)

Controlled variables are those held constant across all experimental conditions
to prevent confounding.

CV1 — Prompt Template.  The same system prompt and language-specific sweep
instructions were used for all models and all runs.  Differences in output can
therefore be attributed to model capability rather than prompt variation.

CV2 — Fixture Set and Error Levels.  All models were evaluated on the same 41
fixtures (16 core + 25 adversarial).  The clean, low, medium, and high error
levels were fixed and identical for all conditions.

CV3 — Inference Parameters.  Temperature (0.2), top_p (0.95), and other
sampling parameters described in Section 3.2.5 were held constant across all
models, with the exception of cloud-specific parameter restrictions noted below.

CV4 — Knowledge-Base Content.  The ChromaDB vector store was built once from
a fixed snapshot of the knowledge-base/ directory and not modified between runs.

CV5 — Evaluation Protocol.  All models were evaluated sequentially through the
same benchmarking script (benchmark.ts) with identical concurrency settings,
retry logic, and scoring functions.

Confounding Variables

Confounding variables are those capable of introducing systematic bias into the
measured outcomes if not acknowledged and mitigated.

Confound 1 — Model Training Data Cutoff.  WCAG 2.2 was published as a W3C
Recommendation in October 2023.  Several of its nine new success criteria
(2.4.11–2.4.13, 2.5.7–2.5.8, 3.2.6, 3.3.7–3.3.8) post-date the knowledge
cutoffs of some models under evaluation.  This creates a systematic disadvantage
for models with earlier cutoffs when detecting the newer SCs.  Mitigation: the
RAG pipeline injects authoritative WCAG 2.2 text at inference time, providing
current guidance regardless of training cutoff.

Confound 2 — Model Provider Throttling and Latency.  Cloud models are served
by external providers whose infrastructure introduces variable latency.  Higher
latency may correlate with deeper reasoning rather than constituting noise, as
larger models generally take longer to respond.  Mitigation: latency is recorded
alongside accuracy metrics; the composite scoring formula (Section 3.5.3)
accounts for response speed explicitly.

Mediating Variable

The quality and completeness of the RAG-retrieved context mediates the
relationship between the RAG condition and F1 score.  Even when RAG is enabled,
if the retrieval query is poorly formed or the relevant knowledge-base chunk is
absent, the model receives no useful grounding.  This mediation is examined
through the hybrid retrieval design (Section 3.3.2) and the five-query strategy
used for HTML fixtures (Section 3.3.3).

Moderating Variable

Model size moderates the effect of both RAG and chain-of-thought reasoning on
detection accuracy.  Smaller models showed larger swings under changes in IV1
and IV2 — for example, qwen3.5:397b (largest model tested) exhibited near-total
recall failure under No-RAG + Think conditions, whilst deepseek-v3.2 (~671B)
remained stable across all four conditions.  This moderating effect is reported
per-model in the ablation results (Section 3.5.2).

Extraneous Variables

Extraneous variables are those that could theoretically affect outcomes but are
judged to have negligible impact in the present study.

Extraneous 1 — Network Jitter.  Momentary fluctuations in cloud API response
times affect the recorded latency values used in composite scoring but are
unlikely to affect the text content of responses.

Extraneous 2 — Cloud API Variability.  As all Phase 1 models were accessed via
cloud APIs, minor differences in provider-side queuing or infrastructure state
could theoretically affect individual run timings.  Because models were tested
sequentially rather than simultaneously, concurrent load contention was
minimised.  The fixed random seed (Section 3.2.5) further ensures that output
variability attributable to sampling noise is controlled across runs.

───────────────────────────────────────────────────────
3.2.3  Methodological Constraints and Boundary Conditions
───────────────────────────────────────────────────────

Several constraints on the scope of the evaluation are acknowledged.

The evaluation is limited to static and dynamic source-code analysis conducted
entirely within the text representation of the file.  The system does not render
pages in a browser, execute JavaScript, or interact with the Document Object
Model at runtime.  Consequently, violations that only manifest under specific
runtime conditions — such as an aria-expanded value that is incorrectly managed
via a complex state machine — may require multiple file passes or developer-
provided context to detect reliably.

The ground-truth fixtures represent a single consistent SaaS-style web
application (a fictional accessibility tooling product named "Nova Health") across
all four languages.  Whilst the fixture size distributions are calibrated to
HTTP Archive median values [1][3], the specific WCAG violation types selected
reflect the categories most commonly found in published large-scale audits [4][5].
The extent to which results generalise to other application types — gaming
interfaces, data dashboards, e-commerce flows — is a limitation that the planned
user study will partially address through exposure to real participant codebases.

The study evaluates cloud-hosted models only in Phases 1 and 2.  Locally hosted
models (which can run entirely offline) were not included in the primary ablation
study.  A preliminary deployment feasibility assessment (Section 3.5.1) tested
four local Ollama-hosted models alongside four cloud equivalents; all four local
models failed with infrastructure connection errors across every condition,
establishing cloud-only deployment as the only viable approach at this scale.
The production extension is configured to route to the highest-performing cloud
model by default; an offline fallback using a smaller local model remains a
future extension of the work.

3.2.4  Technical Scope — Selected LLMs, Vector Database, and API Versions
───────────────────────────────────────────────────────

The following models were evaluated during Phase 1.  All were accessed via the
Ollama local relay, with model names carrying the :cloud suffix indicating cloud-
routed execution.

    gpt-oss:120b-cloud         (approx. 120B parameters)
    kimi-k2.5:cloud            (Moonshot AI)
    kimi-k2.6:cloud            (Moonshot AI, Phase 3 comparison)
    glm-5:cloud                (Zhipu AI GLM-5)
    qwen3.5:397b-cloud         (Alibaba, approx. 397B MoE)
    deepseek-v3.2:cloud        (DeepSeek, approx. 671B MoE)
    minimax-m2:cloud           (MiniMax, approx. 456B MoE)
    minimax-m2.5:cloud
    cogito-2.1:cloud
    nemotron-3-nano:30b-cloud  (NVIDIA, approx. 30B)
    mistral-large-3:675b-cloud
    (plus additional models evaluated in Phase 1 to total 18)

The vector database used throughout is ChromaDB (PersistentClient), version
compatible with Python 3.11.  The embedding model is sentence-transformers/
all-MiniLM-L6-v2, accessed via the chromadb.utils.embedding_functions
SentenceTransformerEmbeddingFunction interface.  BM25 indexing is provided by
the rank_bm25 library (BM25Okapi).

The VS Code extension targets VS Code API version 1.75+ and is implemented in
TypeScript 5 with esbuild as the bundler.  The RAG service is implemented in
Python 3.11 with FastAPI 0.100+ and runs on port 8000.  The Ollama relay is
assumed running on localhost:11434.

All TypeScript evaluation scripts use ts-node and target Node.js 18+.  The
evaluation framework is entirely self-contained: running npm install followed by
npm run bench in any evaluation phase directory is sufficient to reproduce the
benchmarks, provided the Ollama relay and RAG service are active.

───────────────────────────────────────────────────────
3.2.5  Hyperparameter Specification
───────────────────────────────────────────────────────

To ensure a fair and reproducible comparison across all models, a single fixed
set of inference parameters was applied to every model with no model-specific
overrides.  Cloud-routed models receive only the three cloud-safe fields
(temperature, top_p, num_predict); all remaining parameters are silently ignored
by the cloud relay and are documented here for completeness.

Temperature (0.2).  The sampling temperature governs the sharpness of the next-
token probability distribution.  Renze and Guven (2024) conducted a large-scale
empirical study across nine LLMs and five prompt-engineering strategies and found
that temperature values in the range 0.0 to 1.0 produce no statistically
significant difference in problem-solving accuracy, concluding that any value
within this range is equally defensible for analytical tasks [8].  A value of
0.2 was selected to introduce slight determinism appropriate for a classification-
style issue-detection task, whilst remaining well within the operating range of
all model families.

Top-P Nucleus Sampling Threshold (0.95).  Holtzman et al. (2020) introduced
nucleus (top-p) sampling and demonstrated that a threshold of p = 0.95 best
truncates the unreliable low-probability tail of the token distribution whilst
preserving sufficient response diversity [6].  This value aligns with the API
defaults of Google (Gemini) and DeepSeek and is within 0.05 of the defaults used
by OpenAI, Anthropic, Mistral, and Qwen, making it the fairest common baseline
across the heterogeneous set of providers evaluated.

Top-K (k = 40).  Fan et al. (2018) introduced top-k filtering and demonstrated
that restricting sampling to the k most probable tokens reduces incoherence in
generated text [7].  The value k = 40 has since been adopted as the standard
default for focused non-creative generation tasks by major open-source inference
frameworks including Ollama and HuggingFace Transformers, and was used in code-
generation evaluations by Chen et al. (2021) [11].

Repetition Penalty (repeat_penalty = 1.1; frequency_penalty = 0.0;
presence_penalty = 0.0).  Welleck et al. (2020) demonstrated that standard
likelihood-based decoding leads to degenerate repetition [10]; Meister et al.
(2023) showed that top-k and nucleus sampling remain susceptible [9].  A mild
penalty of 1.1 was applied.  The value is deliberately conservative because ARIA
attribute names, WCAG SC identifiers, and other domain vocabulary legitimately
recur across multiple distinct issues within a single response.  Frequency and
presence penalties were set to 0.0 to avoid suppressing legitimate high-frequency
technical terms.

Repetition Look-Back Window (repeat_last_n = 128).  This window spans
approximately two to three structured issue blocks, sufficient to detect genuine
within-issue repetition without penalising the same technical term appearing
correctly in a different issue context.

Maximum Output Length (num_predict = 32,000 tokens).  The output ceiling was set
generously to prevent silent mid-response truncation.  Chen et al. (2021)
identified response completeness as a critical factor in LLM evaluation quality,
noting that incomplete outputs cannot be scored correctly [11].  Analysis of the
first pilot benchmark run showed that the worst-performing models were approaching
the prior 20,000-token ceiling, indicating that issues were being lost to
truncation.  The ceiling was raised to 32,000 tokens accordingly.

Random Seed (seed = 42).  A fixed random seed was applied to all local Ollama
model runs to ensure reproducible outputs.  Pineau et al. (2021) identify fixed
random seeds as a fundamental requirement for reproducible machine learning
evaluation [12].  Cloud-routed models do not expose a seed parameter.
